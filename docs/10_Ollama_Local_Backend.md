# [구현 가이드 10] Ollama 로컬 백엔드 도입

## 1. 결론

이 프로젝트는 Ollama 로컬 모델과 잘 맞는다.

가장 자연스러운 구조는 아래다.

```text
Godot client
  -> Local FastAPI backend
    -> Ollama
```

중요한 이유:

- 현재 클라이언트는 이미 `/health`, `/v1/story/turn` 계약으로 설계되어 있다.
- 따라서 모델을 갈아끼우는 문제와 게임 UI 문제를 분리할 수 있다.
- JSON 스키마 보정도 백엔드에서 처리할 수 있다.

## 2. 왜 Godot 직접 호출보다 낫나

Godot가 Ollama를 직접 호출하면 아래가 전부 클라이언트 책임이 된다.

- 프롬프트 구성
- 모델 응답 파싱
- JSON 형식 오류 복구
- 재시도 정책
- 모델 교체 시 설정 관리

이건 UI 클라이언트가 맡기엔 결합이 너무 강하다.

반대로 백엔드 뒤에 두면:

- 모델 교체가 쉬움
- OpenAI 등 다른 공급자로 바꿔도 Godot는 그대로 둠
- 로깅과 디버깅이 쉬움
- schema repair 가능

## 3. 추천 시작 모델

첫 테스트 기준 추천:

1. `qwen2.5:7b`
2. `llama3.1:8b`
3. `qwen2.5:14b`

권장 순서:

1. `qwen2.5:7b`
장점: 한국어 대응과 구조화 출력 안정성이 비교적 좋고, 초기 디버깅에 부담이 적다.

2. `llama3.1:8b`
장점: 범용 대화 품질이 무난하다. 한국어는 프롬프트 보정이 조금 더 필요할 수 있다.

3. `qwen2.5:14b`
장점: 품질과 일관성이 더 좋다. 대신 더 많은 메모리와 속도 비용이 든다.

## 4. 모델 선택 기준

이 프로젝트는 단순 채팅이 아니라 `구조화된 턴 생성`이 핵심이다.

그래서 좋은 모델 기준은 다음이다.

- 한국어 대사 품질
- JSON 형식 준수율
- 인물 말투 유지력
- 자산 후보 안에서만 선택하려는 통제력
- 로컬 PC에서 감당 가능한 속도

## 5. 최소 백엔드 요구사항

- `GET /health` 가 200을 반환할 것
- `POST /v1/story/turn` 가 게임 스키마를 반환할 것
- `content_rating` 을 항상 채울 것
- 잘못된 자산 ID를 보내지 말 것
- 모델 원문이 JSON 형식을 깨면 서버에서 복구할 것

## 6. 단계별 도입 순서

1. `backend/` 예제를 그대로 실행
2. Godot 설정에서 stub 모드 끄기
3. `Backend Base URL = http://127.0.0.1:8000`
4. `Check Backend` 확인
5. 짧은 입력으로 장면 생성 테스트
6. 그 다음에만 프롬프트 품질 조정

## 7. 초기 테스트 프롬프트 팁

초기엔 모델에게 자유 소설을 길게 요구하지 말고 아래 우선순위를 준다.

1. JSON 정확도
2. 자산 선택 정확도
3. 대사 자연스러움
4. 장기 기억

순서를 거꾸로 가면 디버깅이 어려워진다.

## 8. 현실적인 운영 팁

- 처음엔 `temperature` 를 높이지 말 것
- 응답 길이를 짧게 유지할 것
- 모델 원문 로그를 백엔드에 남길 것
- fallback 응답을 남겨서 JSON 실패를 바로 찾을 것

## 9. 추천 방향

현재 프로젝트 기준 최선은:

1. 클라이언트는 그대로 유지
2. `backend/main.py` 같은 최소 서버 추가
3. Ollama 모델을 뒤에 연결
4. 그 후에 세계관/캐릭터 프롬프트 강화

이 방향이 가장 덜 꼬이고, 교체 비용도 가장 낮다.

# [기획서 05] Godot 로컬 런타임 및 성능 최적화

## 1. 시스템 방향 전환
기존 웹 배포 중심 설계 대신, 본 프로젝트는 **Godot 기반 로컬 클라이언트**를 기준으로 재설계합니다.

이 전환의 이유:
- 유저 자산 업로드와 로컬 파일 접근이 쉬움
- 저장/불러오기 신뢰성이 높음
- 성인용 자산 분리 관리가 쉬움
- VN에 필요한 레이어/UI 연출을 네이티브에서 다루기 편함

---

## 2. 권장 런타임 구조

### 2.1 Godot 클라이언트
담당 역할:
- UI 렌더링
- 장면 전환
- 자산 로딩
- 세이브/로드
- 로컬 설정 관리

### 2.2 AI Adapter
담당 역할:
- 프롬프트 조립
- 로컬 모델 서버 요청
- 응답 파싱
- 오류 복구

### 2.3 데이터 계층
담당 역할:
- 자산 메타데이터 인덱싱
- 세이브 저장
- 로그 관리

---

## 3. Godot 내부 모듈 가정
Autoload 또는 매니저 구조로 아래 정도의 분리를 가정합니다.

- `GameState`
- `NarrativeDirector`
- `AssetLibrary`
- `AiClient`
- `SaveManager`
- `AudioManager`
- `SettingsManager`

이 구조면 UI 씬과 시스템 로직을 분리하기 쉽습니다.

---

## 4. AI 연결 방향

### 4.1 기본 전제
- AI는 Godot 내부에 직접 넣기보다 별도 프로세스로 두는 편이 현실적입니다.
- Godot는 결과를 받는 클라이언트 역할에 집중합니다.

### 4.2 추천 연결 방식
- `localhost` REST
- `localhost` WebSocket

### 4.3 이유
- 모델 교체가 쉬움
- Godot와 생성 백엔드를 독립적으로 디버깅 가능
- 스트리밍 출력과 복구 로직 분리 가능

---

## 5. 텍스트 스트리밍 방향
실시간 생성형 VN에서는 긴 응답을 한 번에 받기보다 **부분 출력**이 체감 품질에 유리합니다.

방향:
- 응답 본문은 토큰 또는 문장 단위 스트리밍
- 구조화 JSON은 마지막에 확정본으로 수신

간단한 운영 방식:
1. 먼저 narration/dialogue를 스트리밍
2. 완료 시 최종 JSON 수신
3. 장면/오디오 상태 확정 반영

초기 버전에서는 스트리밍 없이 완성 응답만 받아도 되지만, UX 강화를 위해 확장 여지를 남겨둡니다.

---

## 6. 성능 최적화 포인트

### 6.1 자산 로딩
- 원본 이미지와 별도로 썸네일/미리보기 캐시 생성
- 자주 쓰는 배경과 스프라이트는 선로딩
- CG는 필요 시점 직전 비동기 로딩

### 6.2 메타데이터 인덱싱
- 폴더 스캔 결과를 매 실행마다 전체 재구성하지 않음
- 변경 감지 후 메타데이터 캐시만 업데이트

### 6.3 대화 로그 관리
- 전체 로그는 저장하되, 런타임 메모리에는 최근 구간만 유지

### 6.4 응답 파싱
- 생성기 응답이 길어도, 렌더러는 필요한 필드만 빠르게 분리해 사용

---

## 7. 저장 포맷 방향

### 7.1 V1 권장
- 세이브: JSON
- 자산 메타데이터: JSON
- 설정: JSON

장점:
- 디버깅 쉬움
- 개발 초기 속도 빠름

### 7.2 확장 시 고려
- 대량 자산 인덱싱 또는 검색 최적화가 필요하면 SQLite 도입

---

## 8. 플랫폼 우선순위
- 1차: Windows
- 2차: Linux
- 3차: macOS

이유:
- 로컬 AI 백엔드 연결과 테스트 환경이 비교적 단순
- 성인용 인디 VN 사용자층과도 맞음

---

## 9. 구현 아이디어
- 첫 버전은 Godot 하나로 UI와 로컬 파일 관리를 완성
- AI는 `localhost` API를 통해 연결
- 자산 메타데이터는 수동 태깅 위주
- 성능 이슈는 자산 수천 개 이상부터 본격 대응

즉, 초기에는 복잡한 최적화보다 **명확한 모듈 분리**가 더 중요합니다.

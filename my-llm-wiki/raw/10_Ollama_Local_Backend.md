# [구현 가이드 10] Ollama 로컬 백엔드 도입

## 1. 결론

이 프로젝트는 Ollama 로컬 모델과 잘 맞는다.

가장 자연스러운 구조는 아래다.

```text
Godot client
  -> Local FastAPI backend
    -> Ollama
```

중요한 이유:

- 현재 클라이언트는 이미 `/health`, `/v1/story/turn` 계약으로 설계되어 있다.
- 따라서 모델을 갈아끼우는 문제와 게임 UI 문제를 분리할 수 있다.
- JSON 스키마 보정도 백엔드에서 처리할 수 있다.
- 현재 문서 기준 권장 시작 모델은 `qwen2.5:14b` 이다.

## 2. 왜 Godot 직접 호출보다 낫나

Godot가 Ollama를 직접 호출하면 아래가 전부 클라이언트 책임이 된다.

- 프롬프트 구성
- 모델 응답 파싱
- JSON 형식 오류 복구
- 재시도 정책
- 모델 교체 시 설정 관리

이건 UI 클라이언트가 맡기엔 결합이 너무 강하다.

반대로 백엔드 뒤에 두면:

- 모델 교체가 쉬움
- OpenAI 등 다른 공급자로 바꿔도 Godot는 그대로 둠
- 로깅과 디버깅이 쉬움
- schema repair 가능

## 3. 추천 시작 모델

현재 프로젝트 기준 추천 순서는 아래다.

1. `qwen2.5:14b`
장점: RP 장면에서 감정선, 캐릭터 유지력, 한국어/영어 혼합 대응, 구조화 출력 안정성의 균형이 가장 좋다. 현재 문서상 기준선으로 삼기 좋다.

2. `qwen2.5:7b`
장점: 로컬 PC 자원이 빡빡하거나 반응 속도를 더 우선할 때 현실적인 경량 대안이다. 품질보다는 반복 실험과 저사양 대응에 유리하다.

3. `llama3.1:8b`
장점: 비교용 선택지로는 괜찮다. 다만 현재 프로젝트에서는 한국어 말투와 JSON 안정성 쪽에서 Qwen 계열보다 추가 보정이 필요할 가능성이 있다.

## 4. 권장 운영 프로필

문서 기준 권장 운영 프로필은 아래다.

- `OLLAMA_MODEL=qwen2.5:14b`
- `OLLAMA_NUM_CTX=4096`
- `OLLAMA_NUM_PREDICT=320`
- `OLLAMA_TEMPERATURE=0.4`
- `OLLAMA_TOP_P=0.9`
- `OLLAMA_REPEAT_PENALTY=1.10`
- `OLLAMA_KEEP_ALIVE=20m`
- `OLLAMA_REQUEST_TIMEOUT_SECONDS=40`
- `BACKEND_WARM_MIN_VRAM_RATIO=0.85`

중요:

- `backend/main.py` 와 `.env.example` 의 기본값을 이제 `qwen2.5:14b` 기준으로 맞춘다.
- 공개 API 계약은 그대로 유지하고, 설정 기본값만 14B 프로필로 승격한다.
- 저사양 환경에서는 `OLLAMA_MODEL=qwen2.5:7b` override를 fallback으로 허용한다.

## 5. 모델 선택 기준

이 프로젝트는 단순 채팅이 아니라 `구조화된 턴 생성`이 핵심이다.

그래서 좋은 모델 기준은 다음이다.

- RP 몰입감
- 캐릭터 유지력
- 한국어/영어 혼합 대응
- JSON 형식 준수율
- 자산 후보 안에서만 선택하려는 통제력
- 로컬 PC에서 감당 가능한 속도

## 6. 최소 백엔드 요구사항

- `GET /health` 가 200을 반환할 것
- `POST /v1/story/turn` 가 게임 스키마를 반환할 것
- `content_rating` 을 항상 채울 것
- 잘못된 자산 ID를 보내지 말 것
- 모델 원문이 JSON 형식을 깨면 서버에서 복구할 것

공개 API 계약은 바꾸지 않는다.

## 7. 단계별 도입 순서

1. `backend/` 예제를 그대로 실행
2. `ollama pull qwen2.5:14b` 와 `ollama serve` 준비
3. 기본값 그대로 서버 실행
4. Godot 설정에서 stub 모드 끄기
5. `Backend Base URL = http://127.0.0.1:8000`
6. `Check Backend` 확인
7. 짧은 입력으로 장면 생성 테스트
8. 필요하면 그 다음에만 7B fallback 또는 튜닝 override 적용
9. 마지막에 프롬프트 품질 조정

## 8. 초기 테스트 프롬프트 팁

초기엔 모델에게 자유 소설을 길게 요구하지 말고 아래 우선순위를 준다.

1. JSON 정확도
2. 자산 선택 정확도
3. 캐릭터 유지력
4. 대사 자연스러움

순서를 거꾸로 가면 디버깅이 어려워진다.

## 9. 현실적인 운영 팁

- 처음 기준은 문서 권장값 그대로 두고 시작할 것
- 너무 느리면 먼저 `qwen2.5:7b` 로만 내려서 비교할 것
- 모델 원문 로그를 백엔드에 남길 것
- fallback 응답을 남겨서 JSON 실패를 바로 찾을 것

## 10. 추천 방향

현재 프로젝트 기준 최선은:

1. 클라이언트는 그대로 유지
2. `backend/main.py` 같은 최소 서버 추가
3. Ollama 모델은 우선 `qwen2.5:14b` 기준으로 뒤에 연결
4. 그 후에 세계관/캐릭터 프롬프트 강화
5. VRAM이나 지연이 부담되면 그때 `qwen2.5:7b` fallback 또는 추가 튜닝 검증을 진행

이 방향이 가장 덜 꼬이고, 교체 비용도 가장 낮다.
